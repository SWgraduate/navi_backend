### 1. 언어 및 문체
- **모든 커밋 메시지는 반드시 영어로 작성할 것.**
- **명령문(Imperative mood) 사용:** 제목(Subject)과 본문(Body)은 동사 원형으로 시작하여 누군가에게 명령을 내리는 듯한 형태(예: `add`, `fix`, `update`)로 작성할 것. (`adds`, `added`, `adding` 사용 금지)

### 2. 커밋 메시지 구조
메시지는 크게 제목(Subject), 본문(Body), 꼬리말(Footer) 세 부분으로 나누며, 각 파트는 **빈 줄(Blank line)**로 구분할 것.
- **포맷:** `<type>(<scope>): <subject>`
  `<BLANK LINE>`
  `<body>`
  `<BLANK LINE>`
  `<footer>`

### 3. 타입 (Type) 지정 규칙
제목의 `<type>`은 다음 중 하나를 **반드시** 선택하여 소문자로 작성할 것.
- **feat:** 새로운 기능 추가
- **fix:** 버그 수정
- **docs:** 문서 수정 (README, 주석 등)
- **style:** 코드 포맷팅, 세미콜론 누락 수정 등 코드의 로직에 영향을 주지 않는 변경
- **refactor:** 코드 리팩토링 (기능 추가나 버그 수정이 없는 코드 변경)
- **test:** 테스트 코드 추가 및 수정
- **chore:** 빌드 업무 수정, 패키지 매니저 환경설정 등 (프로덕션 코드 변경 없음)

### 4. 제목 (Subject) 작성 규칙
- **길이 제한:** 제목은 50자를 넘지 않도록 간결하게 작성할 것.
- **소문자 시작:** 첫 글자는 소문자로 시작할 것. (예: `feat(auth): implement session login`)
- **마침표 금지:** 제목 끝에 마침표(`.`)를 붙이지 말 것.
- **스코프(Scope) 활용 (선택 사항):** 변경된 범위를 명확히 하고 싶다면 괄호 안에 스코프를 명시할 것. (예: `fix(parser): ...`, `feat(api): ...`)

### 5. 본문 (Body) 작성 규칙 (선택 사항이나 권장)
- **작성 기준:** 코드가 '어떻게(How)' 변경되었는지가 아니라, **'무엇을(What)'** 그리고 **'왜(Why)'** 변경했는지 설명할 것.
- **길이 제한:** 각 줄은 72자를 넘지 않도록 줄바꿈을 할 것.
- **문단 구분:** 설명이 길어질 경우 적절히 문단을 나누어 가독성을 높일 것.

### 6. 꼬리말 (Footer) 작성 규칙 (선택 사항)
- **이슈 연결:** 관련된 이슈 트래커 ID를 닫거나 참조할 때 사용할 것.
  - (예시) `Resolves: #12`
  - (예시) `See also: #34, #56`
- **파괴적 변경(Breaking Changes) 명시:** 하위 호환성을 깨뜨리는 변경 사항이 있다면 `BREAKING CHANGE:` 로 시작하여 변경 내용을 명시할 것.
